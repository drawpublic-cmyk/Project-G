# 관리데이 MVP 기획안

## 1. 서비스 한 줄 정의

여러 뷰티 시술을 하루 일정으로 묶어 추천하고, 사용자가 한 번의 요청으로 예약 가능 여부 확인과 확정을 받을 수 있는 관리데이 예약 플래너.

## 2. 해결하려는 문제

한국 사용자는 네일, 속눈썹, 피부관리, 눈썹문신, 왁싱, 헤어 등을 하루에 몰아서 끝내고 싶어 한다. 하지만 실제 예약은 다음 문제가 있다.

- 각 샵의 네이버예약 페이지를 따로 열어야 한다.
- 가능한 시간이 서로 맞지 않으면 일정을 다시 짜야 한다.
- 예약 중간에 누군가 먼저 예약하면 전체 코스가 깨진다.
- 시술별 소요시간, 이동시간, 취소 정책을 사용자가 직접 계산해야 한다.
- 네이버예약 공개 API만으로는 모든 샵의 실시간 예약 생성/확정까지 자동화하기 어렵다.

## 3. MVP 핵심 가설

사용자는 “완전 자동 예약” 전 단계라도, 여러 예약을 대신 맞춰주는 관리데이 코스 추천과 예약대행 요청에 충분한 가치를 느낀다.

검증할 질문:

- 사용자는 네이버예약을 여러 번 직접 하는 대신 예약대행을 요청할까?
- 가장 많이 선택되는 관리 조합은 무엇인가?
- 사용자는 예약 확정까지 몇 분 정도 기다릴 수 있을까?
- 예약 불가 시 대체 시간/대체 샵 제안을 받아들일까?
- 보증금 또는 예약대행 수수료를 지불할 의사가 있을까?

## 4. MVP 범위

### 포함 기능

- 지역 선택: 강남역, 성수, 홍대입구, 잠실 등
- 날짜/시작 시간/예산/이동 허용 범위 선택
- 관리 항목 복수 선택
- 추천 코스 자동 생성
- 예상 총 소요시간, 예상 비용, 이동시간 표시
- 샵별 네이버예약 링크 연결
- 한 번에 예약요청
- 예약 불가 시 대체 코스 제안
- 운영자 확인용 예약 요청 상태 관리

### 제외 기능

- 네이버예약 직접 자동 확정 API 연동
- 모든 샵의 실시간 재고 동기화
- 즉시 결제 정산 시스템
- 완전 자동 취소/변경 처리

## 5. “한 번에 예약요청”의 의미

MVP에서 “한 번에 예약요청”은 사용자가 모든 샵을 즉시 예약 완료한다는 뜻이 아니다.  
정확히는 “선택한 관리데이 코스 전체를 하나의 예약 요청 티켓으로 제출하고, 시스템 또는 운영자가 각 샵의 예약 가능 여부를 확인해 확정해주는 것”이다.

사용자 관점:

1. 추천 코스를 확인한다.
2. “한 번에 예약요청”을 누른다.
3. 이름, 연락처, 선호 조건, 대체 가능 시간 범위를 입력한다.
4. 요청이 접수된다.
5. 예약 가능/불가/대체안 알림을 받는다.
6. 사용자가 최종 승인하면 예약이 확정된다.

운영 관점:

1. 예약 요청 티켓이 생성된다.
2. 각 샵의 네이버예약 링크 또는 샵 채널을 통해 가능 시간을 확인한다.
3. 모든 예약이 가능한 경우 사용자에게 확정 요청을 보낸다.
4. 일부 예약이 불가능하면 대체 코스를 생성한다.
5. 사용자가 대체안을 승인하면 운영자가 예약을 완료한다.

## 6. 예약 상태 흐름

예약 요청은 단일 상태가 아니라 코스 전체 상태와 개별 시술 상태를 나눠 관리해야 한다.

### 코스 전체 상태

- `draft`: 사용자가 코스를 보고 있는 상태
- `requested`: 한 번에 예약요청 제출
- `checking`: 운영자 또는 시스템이 예약 가능 여부 확인 중
- `needs_user_choice`: 일부 예약 불가로 사용자 선택 필요
- `confirmable`: 전체 예약 가능, 사용자 최종 승인 필요
- `confirmed`: 모든 예약 확정
- `failed`: 대체안까지 불가
- `cancelled`: 사용자 또는 운영자가 취소

### 개별 시술 상태

- `candidate`: 추천된 후보
- `checking`: 예약 가능 여부 확인 중
- `available`: 예약 가능
- `held`: 임시 확보 또는 운영자 확인 완료
- `unavailable`: 예약 불가
- `alternative_suggested`: 대체 샵/시간 제안됨
- `booked`: 예약 확정
- `cancelled`: 취소

## 7. 예약 불가 상황 반영 방식

가장 중요한 포인트는 “추천 코스”와 “확정 예약”을 명확히 분리하는 것이다.

추천 코스 화면에서는 다음 문구가 필요하다.

> 현재 코스는 추천 일정입니다. 실제 예약 가능 여부 확인 후 확정됩니다.

예약 요청 후에는 각 시술별 상태를 보여준다.

예시:

- 네일: 확인 중
- 속눈썹펌: 예약 가능
- 피부관리: 해당 시간 예약 불가

피부관리가 불가능한 경우 시스템은 세 가지 대체안을 제안한다.

### 대체안 A: 같은 샵, 다른 시간

- 기존: 16:30 결스킨케어
- 대체: 17:20 결스킨케어
- 영향: 전체 종료 시간이 50분 늦어짐

### 대체안 B: 다른 샵, 같은 시간대

- 기존: 16:30 결스킨케어
- 대체: 16:40 온스킨 강남
- 영향: 이동 8분 추가, 가격 12,000원 증가

### 대체안 C: 코스 순서 변경

- 기존: 네일 → 속눈썹펌 → 피부관리
- 대체: 속눈썹펌 → 네일 → 피부관리
- 영향: 총 이동시간 6분 증가, 종료시간 동일

사용자는 이 중 하나를 고르거나 전체 요청을 취소할 수 있다.

## 8. 예약 불가를 줄이는 제품 전략

### 1차 MVP: 예약 가능성 점수

실시간 재고가 없어도 샵별로 “예약 가능성 점수”를 만들 수 있다.

예:

- 최근 같은 요일/시간대 예약 성공률
- 당일 예약 가능 여부
- 운영자가 확인한 최근 빈 슬롯
- 네이버예약 페이지에서 표시되는 예약 가능 패턴
- 샵 응답 속도

추천 알고리즘은 단순히 평점 높은 샵이 아니라 “관리데이 코스 전체가 성공할 확률이 높은 조합”을 우선한다.

### 2차 MVP: 운영자 임시 확인

사용자가 예약요청을 누르면 운영자가 각 샵을 확인한다.  
확인 중 다른 사용자가 먼저 예약할 수 있으므로 사용자에게는 “확정 전” 상태로 보여준다.

문구:

> 아직 예약 확정 전입니다. 가능 시간이 바뀌면 가장 가까운 대체안을 제안드릴게요.

### 3차: 제휴샵 슬롯 확보

반복적으로 많이 팔리는 조합은 제휴샵과 사전에 슬롯을 확보한다.

예:

- 토요일 13:00-18:00 강남 관리데이 슬롯
- 네일 2자리, 속눈썹 2자리, 피부관리 2자리
- 보증금 결제 후 즉시확정

장기적으로는 이 방식이 “진짜 한 번에 예약”에 가장 가깝다.

## 9. 사용자 플로우

1. 앱 진입
2. 지역, 날짜, 시작 시간, 예산 선택
3. 받고 싶은 관리 선택
4. 추천 코스 확인
5. 한 번에 예약요청
6. 연락처와 대체 가능 조건 입력
7. 예약 가능 여부 확인 중
8. 결과 수신
9. 모든 예약 가능하면 최종 승인
10. 일부 불가능하면 대체안 선택
11. 예약 확정 알림 수신

## 10. 운영자 플로우

1. 예약 요청 티켓 생성
2. 요청 상세 확인
3. 시술별 네이버예약 링크 열기
4. 가능 시간 확인
5. 상태 업데이트
6. 불가능한 항목은 대체 후보 선택
7. 사용자에게 확정안 또는 대체안 발송
8. 사용자 승인 후 예약 진행
9. 예약번호/샵 안내/주의사항 발송

## 12. UX에서 꼭 보여줘야 하는 문구

추천 코스 화면:

> 추천 일정이며, 실제 예약 가능 여부 확인 후 확정됩니다.

예약 요청 완료 화면:

> 요청이 접수됐어요. 각 샵의 가능 시간을 확인한 뒤 확정안 또는 대체안을 보내드릴게요.

일부 예약 불가 화면:

> 피부관리 16:30 시간이 마감됐어요. 가장 가까운 대체안을 찾았습니다.

최종 확정 전:

> 아래 일정으로 예약을 확정할까요? 확정 후 취소 정책은 각 샵 기준으로 적용됩니다.

## 13. MVP 성공 지표

- 추천 코스 생성 후 예약요청 전환율
- 예약요청 후 실제 확정률
- 예약 불가 발생률
- 대체안 수락률
- 평균 확정 소요시간
- 카테고리별 예약 성공률
- 사용자 재사용률
- 제휴샵 전환 가능성이 높은 카테고리

## 14. 초기 운영 추천

처음부터 완전 자동화를 목표로 하지 않는다.  
강남역 1개 지역, 네일/속눈썹/피부관리 3개 카테고리로 시작한다.

초기 운영 방식:

- 앱은 코스 추천과 예약요청 접수 담당
- 운영자는 네이버예약/톡톡/전화로 예약 가능 여부 확인
- 예약 불가 시 대체안 2-3개 제안
- 자주 성공하는 샵은 제휴 후보로 분류
- 반복되는 시간대는 자체 슬롯화

## 15. 결론

이 서비스의 핵심은 단순히 여러 예약 링크를 모아주는 것이 아니다.  
핵심 가치는 “여러 예약이 동시에 성립해야 하는 복잡한 일정 문제를 사용자가 직접 풀지 않게 해주는 것”이다.

따라서 MVP에서는 “즉시 확정”보다 “예약 가능 여부 확인과 대체안 제안”이 더 중요하다.  
예약 불가를 실패로 처리하지 않고, 대체 시간/대체 샵/순서 변경으로 다시 코스를 완성하는 것이 제품의 본질이다.
